Introducing OpenShift Migration Advisor

Read this chapter to learn what OpenShift Migration Advisor is, which problems a migration assessment helps you address, who the service is for, and how you reach it in the Red Hat Hybrid Cloud Console.

What OpenShift Migration Advisor is

OpenShift Migration Advisor is a Red Hat Hybrid Cloud Console service that helps you review a VMware vSphere environment when you are considering OpenShift Virtualization as a target. The service produces a migration assessment from data you provide. The assessment highlights infrastructure and workload information, risk or compatibility signals the product can detect, and planning outputs such as sizing context. It is a planning step. It does not migrate virtual machines by itself.

OpenShift Migration Advisor is implemented with a user experience in the console and with services or optional software that collect or process vSphere information. You do not need to be an OpenShift Virtualization administrator to start an assessment, but you do need access to your vSphere environment or to RVTools output, according to the path you choose.

Problems that a migration assessment is meant to help with

Teams moving from VMware to OpenShift Virtualization often need a shared picture of what is running in the estate before they commit to designs and project dates. OpenShift Migration Advisor is intended to help with the following needs.

  • Visibility — Identify which virtual machines and related resources exist in scope and what the inventory looks like in summary form, subject to the collection method and product limits.

  • Sizing context — Get guidance about CPU, memory, and storage scale that supports discussion of a target OpenShift Virtualization or OpenShift environment. Outputs depend on the data you supply and the version of the product, and are not a substitute for a capacity analysis that your organization still wants to run.

  • Readiness signals — Surface items that the product can flag, such as operating system compatibility, health-related signals where available, storage and network context, and other migration considerations the rules engine reports.

What you get is limited by the depth of data collection, the rules available in the product, and the fact that the assessment is a snapshot in time, not a substitute for a full design review in every organization.

Who typically uses the service, and at what scale

The user experience is designed for VMware administrators and IT and platform architects who know their vSphere environment. You should be able to find or export the data the service accepts, and understand what you are authorizing if you use an agent in your environment.

Many organizations with moderate VMware footprints use a self-service assessment to start the conversation. Very large environments may still use the product for a first look, or they may combine it with a deeper engagement through Red Hat or their own practices. A practical limit on how much inventory you should drive through a single self-service flow depends on product support and on your own operational policies. If your estate is at the very large end, validate approach and support with your Red Hat account or partner before you rely on the product as your only method of analysis.

The service does not replace a formal project plan, resourcing, or a signed infrastructure design. It is one input to those decisions.

How you use OpenShift Migration Advisor in the Red Hat Hybrid Cloud Console

OpenShift Migration Advisor is offered as a web experience after you sign in to the Red Hat Hybrid Cloud Console with your Red Hat account. The interface walks you through creating a migration assessment. Prompts, labels, and legal notices live in the product, and they are the source of truth for the exact flow in your build.

To open the service, use the Red Hat Hybrid Cloud Console applications menu or search to find the OpenShift Migration Advisor or migration assessment entry, then follow the on-screen steps. The service may be labeled to reflect your product bundle. Do not bookmark or share internal, nonproduction, or partner-only hosts. Use only the console that your Red Hat subscription or program assigns you.

If your organization does not have access to the Red Hat Hybrid Cloud Console, contact Red Hat or an authorized partner to find out which assessment options apply to you.

Before you start a migration assessment

This chapter lists prerequisites, explains how a file upload and an in VMware agent differ in data handling, and states how you should use planning and sizing outputs. It also compares a RVTools file with an agent based flow at a high level.

Prerequisites to use a migration assessment

You need the following in place before you can complete a useful assessment. Exact checks appear in the console in your build.

  • A Red Hat account and subscription or program that allows access to OpenShift Migration Advisor in the Red Hat Hybrid Cloud Console

  • A supported browser for the Red Hat Hybrid Cloud Console

  • For RVTools — The ability to run RVTools in your vSphere environment and to export a supported spreadsheet format for upload

  • For an agent — A VMware environment where you can import and run a supported OVA (or other supported format), network paths that allow the agent to reach the assessment service when you use connected mode, and credentials or tokens as the product user interface and your security policy require

  • Familiarity with your own change control and with who may approve data leaving your data center, even in anonymized form, if a collection path or optional data sharing is offered

  • OpenShift — You do not have to be an OpenShift Virtualization day-one expert to start an assessment. You need a basic understanding of why you are targeting OpenShift Virtualization and who will own the target design after the assessment

How data handling differs between a file upload and an agent

You can build an assessment in more than one way, depending on what the product in your build supports. A common choice is upload (for example, RVTools) versus a local agent in VMware. Data handling, reporting depth, and whether anything is available to the cloud console can differ.

  • RVTools upload — The upload usually shares a minimal or anonymized set of data with the service, according to the privacy statement and the technical behavior described for your release. The console shows what you are agreeing to.

  • Agent in VMware — A default for many connected-agent flows is to keep detailed inspection data in the local appliance, not to share it with Red Hat unless you turn on a separate, explicit consent. Some recommendations or views may appear only in the Red Hat Hybrid Cloud Console after you opt in, while full per-VM drill down may be available in the local appliance, depending on the product version. Always use the in-product data sharing terms for your environment.

Planning outputs are estimates, not promises

Any suggested cluster size, migration time window, complexity view, or similar output from OpenShift Migration Advisor is an estimation for your planning. It is based on the inventory and rules the product can see, heuristics that your build implements, and limits on how well any automated model can reflect your runbooks, maintenance windows, network reality, and storage performance.

  • Treat numbers as a starting point for design workshops and for comparing scenarios, not as a service level agreement, quote, or guarantee.

  • Update your plan after you have a real target design, a migration path with Migration Toolkit for Virtualization or any other tool you use, and validation in your test environment.

  • If a number looks wrong, validate inputs (for example, a stale RVTools file, or a subset of the estate that the agent can see) before you make project commitments.

Important

Your organization and your Red Hat services team, not the assessment alone, are responsible for final sizing, scheduling, and risk sign-off for a production cutover.

RVTools upload compared with a VMware agent

The product may offer one or both of these paths. Use what your build exposes and what your security model allows.

RVTools spreadsheet

  • When it helps — You already have RVTools, you want a light-touch upload, and you can accept a summary or anonymized view that matches your chosen privacy setting.

  • What you give up — A file alone does not run live queries or deep in-guest checks in the VMware estate. The assessment depth matches what the spreadsheet and product rules can derive.

Assisted agent in vSphere

  • When it helps — You need a richer view of the environment (including views that may be available only on a local agent appliance) and you can deploy software in the VMware side.

  • What you should plan for — Import, credentials, OPA or policy folders if your build requires them, and network paths. Use the product documentation and runbooks for your build for the exact image name and size.

Creating a migration assessment

Follow one of the procedures that matches the data path your OpenShift Migration Advisor build makes available, and read the reference on what the planner or agent can collect. Your release may offer only one of these paths.

Starting an assessment with a RVTools file

Use this path when the console in your OpenShift Migration Advisor build allows you to create an assessment from a supported RVTools export and you are permitted to create that file in vSphere and upload it.

Prerequisites

  • You completed the items in Prerequisites to use a migration assessment and you can generate a supported RVTools file format and version. Unsupported formats are rejected in the product.

Procedure

  1. In the Red Hat Hybrid Cloud Console, open the OpenShift Migration Advisor or migration assessment feature with your Red Hat account, as you would for any other Red Hat Hybrid Cloud Console app.

  2. Start a new migration assessment and select the RVTools or file-based option when the interface offers it. If the option is missing, your product build or your entitlement may not include this path, or the flow may be renamed.

  3. On the workstation where you run RVTools, connect to your vSphere environment, refresh the data you need, and export a spreadsheet in the form that the product help text or validation message describes.

  4. In the Red Hat Hybrid Cloud Console, use the Upload or Choose file action and select your export. Wait until the product confirms that the file passed validation. Fix any errors the console reports, for example by re-exporting from RVTools or by removing unsupported sheets if instructed.

  5. Complete the remaining on-screen questions. Accept any consent or privacy notices the product shows before the assessment is saved or processed. Read these carefully because they are specific to the path you use and to the region in which the service runs.

  6. Open the new assessment in the Red Hat Hybrid Cloud Console and review the status until the result is Ready or a similar terminal state.

  7. If the result shows errors, use the in-product help or the messages next to the failure. Do not attach sensitive exports in email unless your security process allows a secure channel.

Next steps — When the assessment is ready, continue with Interpreting risks and migration considerations in an assessment and the other result topics in this guide. If you are considering an agent later for deeper data, you can also read How data handling differs between a file upload and an agent.

Starting an assessment with a VMware agent

The assisted migration agent connects to the Migration Planner service, collects vSphere data that your rules allow, and reports to the Red Hat Hybrid Cloud Console in supported modes. This procedure describes a typical, high-level sequence. The authoritative steps are the ones you receive when you generate or download the OVA, and the agent and planner documentation for your version.

Prerequisites

  • A VMware vSphere deployment where you can import an OVA and power on a VM that meets the image requirements.

  • Network and firewall settings that let the agent reach the assessment and planner endpoints in the mode you select (connected or disconnected), and any proxy settings your site requires. Your organization must allow these connections under its policies.

  • Identifiers that the service assigns, such as an agent ID and source ID in UUID form, and a folder of OPA policy files if your build requires OPA for VM validation. Use the value placeholders your operator gives you, not examples from a document.

  • Optional JWT or other authentication material if the console in your build requires a token to register the agent with the planner service, generated through the process your organization uses, not with sample keys from a repository.

Procedure at a high level

  1. In the Red Hat Hybrid Cloud Console, start a new migration assessment and choose the agent-based path when the interface offers it, or use the Generate or Download agent action if your product presents it before an assessment is created, depending on your build.

  2. Record the UUIDs and any registration steps the console or planner gives you. Store secrets in a password safe, not in plain text in tickets or wikis.

  3. Import the OVA into the VMware cluster your design specifies. Allocate CPU, memory, and disk to match the release notes for the agent image you are using, not a generic size.

  4. Configure the agent to run in the mode your design requires, connected or disconnected, and point it to the console and planner endpoints the product user interface or your Red Hat operator specifies for your environment only. Do not copy internal host names from a lab into production.

  5. Start the agent and confirm in the Red Hat Hybrid Cloud Console that the source moves to a healthy, reporting state, using the status the product displays.

  6. When the assessment is complete, open the results in the Red Hat Hybrid Cloud Console. If your build also provides a local appliance with a report view, use that for any detailed drill-down the product team documents for that path.

Next steps — Review the assessment outputs as described in Interpreting risks and migration considerations in an assessment. For command lines, flags, and version-specific deployment details, use the Red Hat Migration Planner and assisted migration agent documentation for your product version, not a generic clone of upstream examples.

Important

Example commands in open source repositories are technical references only. Use your own tokens, UUIDs, and endpoints, and never post credentials or real identifiers in a public or shared log.

Categories of vSphere data the assessment can use

The Migration Planner service and the assisted migration agent are built to work with a defined set of vSphere and workload signals. A RVTools file carries a subset on the spreadsheet. A live agent can collect a broader or fresher view, depending on the version, mode, and your consent settings.

Summary of categories (exact fields depend on the release and the path you use)

  • Datastores — Free and total capacity, and storage or protocol type, when the data source provides them.

  • Hosts — Count and high-level state where exposed.

  • Networks — Type and network identifiers, such as VLAN, when the export includes them.

  • Clusters and datacenters — Counts in summary form.

  • General inventory — A vCenter or inventory scope identifier the assessment uses to group the run.

  • Virtual machines — In aggregate, counts and sizing, including virtual CPUs, memory, virtual disk count and size, and guest operating system where the collector can read it, plus a status or health signal. The product applies rules to decide whether a VM is migratable in a supported sense, or should be reviewed manually, according to the policies installed for your run.

Note

This list reflects the categories the upstream migration-planner project documents for assessment design. A specific OpenShift Migration Advisor build or subscription might expose only part of a category, or label fields with different names in the console. Use the in-product help and data-handling information in your release as the final reference.

Do not paste a full RVTools workbook or a raw export into any channel that is not approved for confidential infrastructure data.

Understanding the assessment

This chapter helps you read risk and readiness information, capacity style outputs, and, when your build offers them, time and complexity style views.

Interpreting risks and migration considerations in an assessment

When a migration assessment is ready, the Red Hat Hybrid Cloud Console and, where your build offers it, a local appliance, show indications the product can derive from the data. These may be labeled as risks, blockers, warnings, or considerations depending on the release.

  • What the result is — A machine-driven view of items that match rules in the product and, where your version uses them, in optional policy engines such as OPA for virtual machine validation. The report does not replace a full human review of application behavior or of data and compliance needs.

  • What you should do — Triage each line with the owner of the workload and of the dependencies it has. If the user interface links to a remediation note or a Red Hat document, use that as the source for the pattern the product detected, not a third-party copy.

  • What the result is not — The assessment is not a warranty that every workload will run on OpenShift Virtualization or OpenShift without design or change work, or that all edge cases are covered.

Treating a message as a formal security or compliance finding is a decision for your own governance process in addition to the tool output.

Sizing and capacity-related outputs

A migration assessment can include sizing guidance for a target OpenShift or OpenShift Virtualization footprint, in the form the product offers in your version. The service compares the aggregate resource picture from your RVTools file or agent run to the overhead and planning assumptions the product team builds into that release. It is not a substitute for measuring real utilization and storage performance on the cluster you eventually build, or for your resiliency and high-availability design.

  • How to use the output — Compare the suggested size to your own rules for headroom, maintenance windows, and separate failure domains. If you split work across many OpenShift projects or namespaces, you may need more nodes or pools than a single top-level line item shows.

  • When to dig deeper — If the report flags very uneven resource use, unusual storage or network patterns, or legacy guests that are hard to size from inventory alone, treat the sizing line as a conversation starter, not a purchase order.

Final hardware or cloud shape and storage classes are decisions you make at detailed design, with your Red Hat account or services team and your own internal review when you add them.

Migration time and complexity in the product

Some OpenShift Migration Advisor builds may show a migration time estimate, a complexity view, or other planning hints next to sizing and risk information. The exact labels and controls come from the console. Do not treat a time range or a score as a service level agreement, a project deadline, or a full picture of work for every application team.

Important

Migration time and complexity style outputs are partial or still evolving in many builds. They are not confirmed in the same way as core inventory and risk views. The product may add, move, or hide them between updates. Your product release notes and the help in the product for your entitlement are the source of truth. If a field is missing, experimental, or labeled as beta in the user interface, do not use it for contractual planning.

When you do see a time or complexity line and you use it only for internal order-of-magnitude planning, still compare the result to your runbooks, maintenance windows, and application dependencies. The model does not know blackout dates or data migration work outside the scope of the inventory.

After the assessment

This chapter places OpenShift Migration Advisor next to Migration Toolkit for Virtualization in a migration program, and suggests clear next steps once you have a result. It does not include partner or third party marketplaces.

How OpenShift Migration Advisor relates to Migration Toolkit for Virtualization in a migration program

OpenShift Migration Advisor and Migration Toolkit for Virtualization have different jobs in a typical program that moves virtual machines from VMware to OpenShift Virtualization. OpenShift Migration Advisor is for a pre-migration assessment and planning view of the estate. Migration Toolkit for Virtualization is the Red Hat product line for executing the migration and managing the movement of virtual machines after you have a design and a target cluster that your organization has approved.

  • Use OpenShift Migration Advisor when you need a structured inventory and readiness view and planning outputs from a supported RVTools file or assisted agent run, as your build of the product allows.

  • Use Migration Toolkit for Virtualization when you are ready to create migration plans and to run cutover in line with the Migration Toolkit for Virtualization product documentation and your own change and security process.

A fully automated hand over from a finished OpenShift Migration Advisor result into a Migration Toolkit for Virtualization plan is not guaranteed in every Red Hat offering, OpenShift and OpenShift Virtualization version, or subscription. Read the OpenShift Migration Advisor, OpenShift Virtualization, and Migration Toolkit for Virtualization product documentation for your environment and use only the integrated workflows that Red Hat documents for your release line up.

What to do after the assessment

A migration assessment is an input to a larger program, not the end of the work. When you are satisfied with the data you submitted and the result in the Red Hat Hybrid Cloud Console, you can do the following with your own organization and, where you have a contract, with Red Hat.

  • Align owners — Share the result with the platform, application, and security owners who will sign off on a target design.

  • Close gaps — Address any high severity items the report lists before you invest in a full Migration Toolkit for Virtualization cutover plan. Some items will need a human decision even when the tool cannot automate a fix.

  • Design the target — Build a final OpenShift and OpenShift Virtualization design that includes networks, storage, identity, and day two operations. The assessment supports that design, it does not replace it.

  • Engage Red Hat — When you have a Red Hat account or support contract, use the support and services paths your account team documents. This user guide does not list internal contact details or nonpublic programs.

Frequently asked questions

Short answers to common questions about how OpenShift Migration Advisor is delivered, which languages are supported, and how the service relates to very large vSphere environments.

How often OpenShift Migration Advisor changes

OpenShift Migration Advisor is developed as a web service on the Red Hat Hybrid Cloud Console. Ship cadence mixes larger increments with frequent updates to the live experience.

OpenShift Migration Advisor grows continuously as a web project. Larger features are typically delivered on a monthly sprint rhythm. You should also expect smaller features and fixes to appear on a weekly cadence as the service is updated.

Language availability for the product and this guide

The console experience and this user guide are published for English-speaking users today. Other locales may be considered as the product matures.

Today, the OpenShift Migration Advisor user interface and this guide are available in English. Future releases may broaden regional or language coverage; any change will be announced through normal product communication channels.

Using OpenShift Migration Advisor in very large VMware estates

The user experience is tuned for common midsize footprints. Larger organizations can still use the service for an early view, within product limits and your own policies.

OpenShift Migration Advisor is often used by organizations whose VMware footprint is in a midsize range on average, including environments on the order of two thousand virtual machines. That focus does not block you from uploading inventory from a larger estate. If you operate at very large scale, you can still use OpenShift Migration Advisor for initial investigation and assessment, and you should confirm limits, support expectations, and operational approach with your Red Hat account team or partner where applicable.

Glossary and additional resources

This chapter defines terms used in this user guide and lists public kubev2v repositories for readers who work with source at a high level. It does not replace the Red Hat product documentation for a given release.

Glossary of terms for this guide

Short definitions to align language in the rest of this user guide. Product names and feature labels in the live console are authoritative for a given release.

  • Migration assessment — A OpenShift Migration Advisor result that is built from a supported data set and the rules and analytics in the product. It is a planning view, not a live migration.

  • Migration Planner — The Red Hat service side of the solution that stores and presents assessments and may be involved in generating the agent for VMware. The open source repository name is a development reference.

  • Assisted migration agent — Software, often as an OVA or image for VMware, that you run in or alongside vSphere to send inventory to the assessment service, according to the mode and consent you select.

  • RVTools — A tool that exports vSphere data to a spreadsheet. OpenShift Migration Advisor can accept a supported RVTools file when that path is enabled in your product build.

  • Migration Toolkit for Virtualization — The Red Hat software used to execute migration of virtual machines to OpenShift Virtualization. It is for migration execution, not a substitute for a pre-migration assessment, though you may use both in the same program of work.

Open source projects that implement the assessment stack

OpenShift Migration Advisor on the Red Hat Hybrid Cloud Console is a Red Hat product. The kubev2v community on GitHub publishes source code for related components. Red Hat support, scope, and build identifiers follow your subscription and your product version, not a public repo alone.

  • migration-planner — Migration Planner service in the kubev2v organization.

  • migration-planner-ui-app — Console user interface in the kubev2v organization.

  • assisted-migration-agent — Assisted migration agent in the kubev2v organization.

If you open a public issue or a fork, remove host names, account identifiers, and credentials from logs or command output before you post, in line with your information security process.
