Documentation for the OpenShift migration advisor (OMA), a Red Hat Hybrid Cloud Console service for planning migration from VMware vSphere toward Red Hat OpenShift. Scope includes prerequisites, generating an automated migration report from an RVTools export or assisted migration agent, data-handling and estimate limits, interpreting report outputs and next steps, positioning relative to Migration Toolkit for Virtualization, reference FAQs and glossary material, and related kubev2v repositories.

Introducing the OpenShift migration advisor

Read this chapter for an overview of the OpenShift migration advisor, which problems an automated migration report helps you address, who the service is for, and how you reach it in the Red Hat Hybrid Cloud Console.

Overview of the OpenShift migration advisor

The OpenShift migration advisor is a Red Hat Hybrid Cloud Console service designed to help you evaluate VMware vSphere environments when considering Red Hat OpenShift Container Platform (specifically OpenShift Virtualization) as a migration target.

By analyzing data that you provide, the service generates an automated migration report that highlights infrastructure and workload details, identifies risk or compatibility signals, and provides planning outputs such as sizing context. It is strictly a planning tool and does not migrate virtual machines itself.

Key features and implementation

  • Access — The advisor runs as a web service within the Red Hat Hybrid Cloud Console.

  • Data collection — You can provide data by using either an RVTools export in a supported format (upload) or an assisted migration agent deployed directly in your VMware environment.

  • Requirements — You do not need to be a Red Hat OpenShift administrator to generate a report, but you must have access to your vSphere environment or existing RVTools output, depending on your chosen path.

Problems that an automated migration report is meant to help with

Teams moving from VMware to Red Hat OpenShift often need a comprehensive overview of what is running in their environment before they commit to designs and project dates. The OpenShift migration advisor is intended to help with the following needs:

  • Visibility — Identify which virtual machines and related resources exist in scope and what the inventory looks like in summary form, subject to the collection method and product limits.

  • Sizing context — Get guidance about CPU, memory, and storage scale that supports discussion of a target OpenShift Virtualization or OpenShift environment. Outputs depend on the data you supply and the version of the product, and are not a substitute for a capacity analysis that your organization still wants to run.

  • Readiness signals — Surface items that the product can flag, such as operating system compatibility, health-related signals where available, storage and network context, and other migration considerations the rules engine reports.

The assessment is limited by the depth of data collection and the rules available in the product. The automated report is a snapshot in time and does not replace a full design review.

Who typically uses the service, and at what scale

This service is intended for VMware administrators, IT architects, and platform architects who are familiar with their vSphere environments. You can find or export the data that the service accepts, and you must understand what you are authorizing when you use an agent in your environment.

Organizations with moderate VMware environments can use a self-service assessment for initial analysis. Very large environments can combine this assessment with a comprehensive analysis through Red Hat or internal practices. Practical limits on inventory volume per self-service flow depend on product capabilities and your organization’s operational policies.

The service does not replace a formal project plan, resource planning, or a signed infrastructure design. It is only one input to those decisions.

How you use the OpenShift migration advisor in the Red Hat Hybrid Cloud Console

The Red Hat OpenShift migration advisor is offered as a no-cost web service after you sign in to the Red Hat Hybrid Cloud Console with your Red Hat account. The interface guides you through creating an automated migration report. The prompts, labels, and legal notices in the product are the authoritative source for the exact process in your environment.

To open the service, use the Red Hat Hybrid Cloud Console applications menu or search to find the OpenShift migration advisor or the automated migration report entry, and then follow the on-screen steps. The service might be labeled to reflect your product bundle. Do not bookmark or share internal, nonproduction, or partner-only URLs. Use only the console that your Red Hat subscription or program assigns you.

If your organization does not have access to the Red Hat Hybrid Cloud Console, contact Red Hat or a Red Hat partner to determine which automated report options apply to you.

Before you start an automated migration report

Prerequisites to use an automated migration report

You need the following in place before you can complete a useful automated report. Exact checks appear in the console in your build.

  • A Red Hat account with access to the OpenShift migration advisor in the Red Hat Hybrid Cloud Console. Your organization must have an entitlement through a Red Hat subscription or another eligible program.

  • A supported web browser to load the Red Hat Hybrid Cloud Console. For details about web browser requirements, see the Browser Support link at the bottom of the Hybrid Cloud Console landing page.

  • For RVTools — The ability to run RVTools in your vSphere environment and to export an RVTools spreadsheet in a supported format for upload.

  • For an agent — A VMware vSphere deployment where you can import the assisted migration agent OVA and power on a VM that meets the image requirements for your release, as described in Starting an automated migration report with a VMware agent, network paths that allow the agent to reach the reporting service when you use connected mode, and credentials or tokens as the product user interface and your security policy requirements

  • Familiarity with your own change control and with who may approve data leaving your data center, even in anonymized form, if a collection path or optional data sharing is offered

  • OpenShift — You do not have to be an OpenShift Virtualization day-one expert to start an automated report. You need a basic understanding of why you are targeting OpenShift Virtualization and who will own the target design after the automated report

How data handling differs between a file upload and an agent

You can build an automated report in more than one way, depending on what the product in your build supports. A common choice is upload (for example, RVTools) versus a local agent in VMware. Data handling, reporting depth, and whether anything is available to the cloud console can differ.

  • RVTools upload — The upload usually shares a minimal or anonymized set of data with the service, according to the privacy statement and the technical behavior described for your release. The console shows what you are agreeing to.

  • Agent in VMware — A default for many connected-agent flows is to keep detailed inspection data in the local appliance, not to share it with Red Hat unless you turn on a separate, explicit consent. Some recommendations or views may appear only in the Red Hat Hybrid Cloud Console after you opt in, while full per-VM drill down may be available in the local appliance, depending on the product version. Always use the in-product data sharing terms for your environment.

Planning outputs are estimates, not promises

Any suggested cluster size, migration time window, complexity view, or similar output from the OpenShift migration advisor is an estimate for your planning. It is based on the inventory and rules the product can see, heuristics that your build implements, and limits on how well any automated model can reflect your runbooks, maintenance windows, network reality, and storage performance.

  • Treat numbers as a starting point for design workshops and for comparing scenarios, not as a service level agreement, quote, or guarantee.

  • Update your plan after you have a real target design, a migration path with Migration Toolkit for Virtualization or any other tool you use, and validation in your test environment.

  • If a number appears incorrect, validate your inputs (for example, a stale RVTools file or a subset of the environment that the agent detects) before you make project commitments.

Important

Your organization and your Red Hat services team, not the automated report alone, are responsible for final sizing, scheduling, and risk sign-off for a production cutover.

Creating an automated migration report

Follow one of the procedures that matches the data path your OpenShift migration advisor build makes available, and read the reference on what the planner or agent can collect. Your release may offer only one of these paths.

About RVTools for automated migration reports

RVTools is a free utility from Dell Technologies that exports VMware vSphere data to a spreadsheet. The OpenShift migration advisor can accept an RVTools file in a supported format when your product build enables the file-based path for an automated migration report.

RVTools is a Windows application for auditing and reporting on virtual machines, clusters, datastores, and related inventory in your VMware environment. You generate the spreadsheet export outside the OpenShift migration advisor.

When your build exposes the RVTools path, you can upload an RVTools export in a supported format and version in the Red Hat Hybrid Cloud Console. Exports that do not match what the product validates are rejected.

For the official RVTools download, documentation, and support options, see RVTools on Dell Technologies.

For a guided walkthrough of evaluating VMware migration readiness with the OpenShift migration advisor using a file-based path, see Evaluate VMware migration-readiness with the OpenShift migration advisor (Red Hat Arcade).

Starting an automated migration report with an RVTools file

Use this path when you can create an automated report from a supported RVTools export in the OpenShift migration advisor console, and you have permission to create that file in vSphere and upload it.

Note

For a hands-on Arcade demo of this flow, see Evaluate VMware migration-readiness with the OpenShift migration advisor.
Prerequisites

  • You have completed the prerequisites for using an automated migration report, and you can generate a supported RVTools file format and version. The service rejects unsupported formats.

Procedure

  1. In the Red Hat Hybrid Cloud Console, sign in with your Red Hat account and open the OpenShift migration advisor or the automated migration report feature.

  2. Start a new automated migration report and select the RVTools or file-based option if it is available. If the option is missing, your environment or your entitlement might not include this path, or the flow might be renamed.

  3. On the workstation where you run RVTools, connect to your vSphere environment, refresh the data that you need, and export a spreadsheet in the format that is described by the product help text or validation messages.

  4. In the Red Hat Hybrid Cloud Console, click Upload or Choose file and select your export. Wait until the file passes validation. Fix any errors that are reported in the console, for example, by re-exporting from RVTools or by removing unsupported sheets if instructed.

  5. Complete the remaining on-screen questions. Accept any consent or privacy notices that are displayed before the automated report is saved or processed. Read these carefully because they are specific to the path that you use and to the region in which the service runs.

  6. Open the new automated report in the Red Hat Hybrid Cloud Console and review the status until the result is Ready or a similar completed state.

  7. If errors are displayed, use the in-product help or the messages next to the failure. Do not attach sensitive exports in an email unless your security process provides a secure channel.

Next steps

  • After the automated report is ready, see Interpreting risks and migration considerations in an automated migration report and the other result topics in this guide. If you are considering an agent later for more detailed data, see How data handling differs between a file upload and an agent.

Starting an automated migration report with a VMware agent

The assisted migration agent connects to the Migration Planner service, collects vSphere data according to your configured policies, and sends data to the Red Hat Hybrid Cloud Console in supported modes. The following steps provide a typical, high-level sequence. For the authoritative steps, follow the instructions you receive when you generate or download the OVA, and see the agent and planner documentation for your version.

Note

For a hands-on Arcade demo of deploying and using the OVA agent with the OpenShift migration advisor, see The OpenShift migration advisor using OVA Agent.
Prerequisites

  • A VMware vSphere deployment where you can import an OVA and power on a VM that meets the image requirements.

  • Network and firewall settings that enable the agent to connect to the reporting and planner endpoints in the mode that you select (connected or disconnected), and any proxy settings that your site requires. Your organization must allow these connections under its policies.

  • Identifiers that the service assigns, such as an agent ID and source ID in UUID format, and a folder of OPA policy files if your environment requires OPA for VM validation. Use the value placeholders that your operator provides, not examples from a document.

  • Optional JWT or other authentication material if the console in your environment requires a token to register the agent with the planner service. This material must be generated through the process that your organization uses, not with sample keys from a repository.

Procedure

  1. In the Red Hat Hybrid Cloud Console, start a new automated migration report and choose the agent-based path if it is available. Alternatively, click Generate or Download agent if either option is displayed before an automated report is created, depending on your environment.

  2. Record the UUIDs and any registration steps that are provided. Store secrets in a password safe, not in plain text in tickets or wikis.

  3. Import the OVA into the VMware cluster that your design specifies. Allocate CPU, memory, and disk to match the release notes for the agent image that you are using, not a generic size.

  4. Configure the agent to run in the mode that your design requires, connected or disconnected, and point it to the console and planner endpoints that are specified in the product user interface or by your Red Hat operator for your environment only. Do not copy internal hostnames from a lab into production.

  5. Start the agent and confirm in the Red Hat Hybrid Cloud Console that the source moves to a healthy, reporting state by using the status that is displayed.

  6. After the automated report is complete, open the results in the Red Hat Hybrid Cloud Console. If your environment also provides a local appliance with a report view, use that for any detailed views that are documented for that path.

Next steps

  • Review the automated report outputs as described in Interpreting risks and migration considerations in an automated migration report. For command lines, flags, and version-specific deployment details, see the Red Hat Migration Planner and assisted migration agent documentation for your product version, not generic upstream examples.

Important

Example commands in open-source repositories are technical references only. Use your own tokens, UUIDs, and endpoints. Never post credentials or real identifiers in a public or shared log.

Categories of vSphere data the automated report can use

The Migration Planner service and the assisted migration agent are built to work with a defined set of vSphere and workload signals. An RVTools file contains a subset of this data in spreadsheet form. A live agent can collect a broader or fresher view, depending on the version, mode, and your consent settings.

Summary of categories (exact fields depend on the release and the path you use)

  • Datastores — Free and total capacity, and storage or protocol type, when the data source provides them.

  • Hosts — Count and high-level state where exposed.

  • Networks — Type and network identifiers, such as VLAN, when the export includes them.

  • Clusters and datacenters — Counts in summary form.

  • General inventory — A vCenter or inventory scope identifier the automated report uses to associate the collection run.

  • Virtual machines — In aggregate, counts and sizing, including virtual CPUs, memory, virtual disk count and size, and guest operating system where the collector can read it, plus a status or health signal. The product applies rules to decide whether a VM is supported for migration, or should be reviewed manually, according to the policies installed for your run.

Note

This list reflects the categories the upstream migration-planner project documents for automated report design. A specific OpenShift migration advisor build or subscription might expose only part of a category, or label fields with different names in the console. Use the in-product help and data-handling information in your release as the final reference.

Do not paste an RVTools workbook in full or a raw export into any channel that is not approved for confidential infrastructure data.

What the agent can assess

The assisted migration agent goes beyond static inventory to classify virtual machines for migration readiness and produce planning outputs such as complexity scores and time estimates. This chapter explains how those outputs are derived so you can interpret and act on them effectively.

How the agent classifies virtual machines

The assisted migration agent evaluates each virtual machine in your vSphere inventory and assigns it to one of three readiness categories. These categories appear in the automated migration report and help you prioritize remediation work before you plan a migration cutover.

Migratable

The virtual machine passes all rules the product applies. No blockers or warnings are detected.

Migratable with warnings

The virtual machine can be migrated, but the agent has detected conditions you should review and resolve before or during the migration. Common warnings include:

  • Changed Block Tracking (CBT) not enabled — CBT improves incremental migration efficiency. Enable it on the VM in vSphere before migration.

  • VMware Tools not installed or outdated — VMware Tools must be installed and at a supported version for a reliable migration.

  • More than 4 NICs attached — Virtual machines with a high NIC count require additional network configuration planning on the target.

Warnings do not prevent migration, but they increase the risk of issues during cutover if left unaddressed.

Not migratable

The virtual machine has at least one condition that blocks migration based on the product rules. Common blockers include:

  • Unsupported disk types — Raw Device Mappings (RDM) and shared VMDKs are not supported by the migration toolchain.

  • Unsupported guest operating system — The OS has not been verified for conversion by the underlying migration toolchain.

  • Snapshots present — Consolidate or remove all snapshots before migration.

  • USB devices attached — Virtual USB devices are not supported and must be removed before migration.

  • NVRAM encryption — VMs with NVRAM encryption require additional handling before migration is possible.

When a VM is marked as not migratable, the report shows the specific reason. Address each blocker before including the VM in a migration plan.

Note

Classification rules depend on the version of the product and the policies installed in your environment. Use the in-product help as the final reference for your release.

How complexity scores work

When your OpenShift migration advisor build displays a complexity view, each virtual machine or workload group is assigned a score from 1 to 4. The score is derived from two independent dimensions: the guest operating system type and the total provisioned disk size. A higher score indicates a more complex migration that may need more preparation or manual handling.

OS complexity scores

The product determines the OS score by matching the guest OS name that VMware reports against a defined list. Matching is case-insensitive. If the OS name does not match any known entry, the VM receives a score of 0, which means complexity cannot be assessed automatically.

Score Meaning Examples

1

Straightforward migration, minimal intervention expected

Red Hat Enterprise Linux 6–10, Fedora, CentOS 6–9, Oracle Linux 6–9, Rocky Linux 8–9

2

Standard migration, some attention required

Red Hat Enterprise Linux 4–5, SUSE Linux Enterprise 12–15, Windows Server 2016–2025, Windows 10 and 11

3

More complex migration, additional effort expected

Ubuntu, Debian, SUSE Linux Enterprise 8–11, openSUSE, AlmaLinux 8–9, older Windows Server (2000–2012 R2), Windows XP–8, Oracle Solaris, FreeBSD

4

May require manual intervention or special handling

Database workloads such as Microsoft SQL Server

0

OS not recognized; complexity cannot be assessed automatically

Unknown or custom operating systems

Disk complexity scores

The disk score is based on the total provisioned disk size across the virtual machine.

Score Total provisioned disk size Meaning

1

≤ 10 TB

Small to moderate storage; standard migration path

2

11–20 TB

Larger storage; plan transfer time carefully

3

21–50 TB

Significant storage; time estimates and throughput planning are important

4

> 50 TB

Very large storage; requires detailed migration scheduling and throughput benchmarking

How to use complexity scores

Use complexity scores to prioritize and sequence your migration work:

  • Start with score-1 VMs to build confidence and validate your target environment before tackling more complex workloads.

  • Review score-3 and score-4 VMs with your application owners early, as they are likely to need design decisions, extra testing, or manual steps that are not visible from inventory alone.

  • A score-0 OS is a signal to verify the guest OS manually before including the VM in any migration batch.

Note

Complexity scores are planning inputs, not migration results. They indicate the relative level of effort typically associated with that OS type or disk size, not whether a migration will succeed.

How migration time estimates are calculated

When your OpenShift migration advisor build displays a migration time estimate, the value is the sum of two modelled phases: storage data transfer and post-migration troubleshooting. Understanding the assumptions behind each phase helps you interpret the estimate and adjust inputs to better reflect your environment.

Storage migration phase

The storage phase estimates the time required to copy VM disk data from the source VMware datastore to the target cluster over the network.

The default assumptions are:

  • Transfer rate: 620 Mbps (approximately 77.5 MB/s), equivalent to roughly 110 minutes per 500 GB of provisioned data.

  • Data size: Total provisioned disk size across all VMs in scope, measured in GB.

  • Transfer model: Linear sustained transfer; the estimate does not account for concurrent migrations running in parallel.

If your network has higher or lower throughput than the default, the resulting estimate may differ significantly from real migration time. Use the Migration Forecaster, where available in your build, to measure real datastore-to-datastore throughput. See Benchmarking storage throughput with the Migration Forecaster.

Post-migration troubleshooting phase

The troubleshooting phase estimates the calendar time your team spends on post-migration checks and remediation after VMs are moved.

The default assumptions are:

  • Time per VM: 60 minutes of engineer time per VM.

  • Team size: 10 engineers working in parallel.

  • Working hours: 8 hours per day.

The model divides total engineer-minutes by the number of engineers to produce wall-clock time. For example, 50 VMs at 60 minutes each with 10 engineers working in parallel gives a 5-hour window.

Combined estimate

The total estimated migration time is the sum of the storage phase and the troubleshooting phase. The product presents this as a range, not a fixed commitment.

Important

Migration time estimates are planning inputs for internal order-of-magnitude discussions. They do not account for application-specific blackout windows, data migration work outside VM storage, or dependencies between workloads. Compare the estimate against your runbooks and team capacity before using it for scheduling.

Benchmarking storage throughput with the Migration Forecaster

The Migration Forecaster measures actual disk-copy throughput between VMware datastore pairs in your vCenter environment. Where your OpenShift migration advisor build includes this feature, you can use real throughput measurements instead of the default transfer rate assumption to produce more accurate migration time estimates.

How the Migration Forecaster works

The forecaster runs disk-copy benchmarks directly on your vCenter infrastructure:

  1. It creates a temporary virtual machine and a benchmark disk on the source datastore.

  2. It copies the benchmark disk to the target datastore multiple times and measures wall-clock transfer time for each iteration.

  3. It computes throughput statistics—including mean, median, and confidence intervals—from the measured results.

  4. It uses those statistics to project a data-driven migration time estimate for each datastore pair.

All temporary resources are cleaned up automatically when each benchmark completes, whether or not it succeeds.

Temporary resources created in vCenter

The forecaster creates the following resources in your vCenter environment while a benchmark is running:

Resource Purpose

Filler VM (Alpine Linux, ~256 MB)

Fills the benchmark disk with random data so the copy reflects realistic conditions. Destroyed when filling completes.

Alpine boot disk and seed ISO

Operating system and configuration for the filler VM. Deleted after each datastore pair completes.

Benchmark disk (default 10 GB)

The disk filled with random data that the forecaster copies to the target. Deleted after each pair completes.

Clone disk on target datastore

Copy of the benchmark disk created on the target during each iteration. Deleted after each iteration.

Temporary resource names are prefixed with forecaster- or filler-image- so they are easy to identify if manual cleanup is needed.

Note

If the agent stops while a benchmark is in progress, temporary files or directories may remain on the datastores and require manual cleanup.

Required vSphere privileges

The vCenter credentials used for benchmarking must include the following privileges. The forecaster verifies these before it starts and reports any that are missing.

  • Datastore.AllocateSpace

  • Datastore.Browse

  • Datastore.DeleteFile

  • Datastore.FileManagement

  • VirtualMachine.Inventory.Create

  • VirtualMachine.Inventory.Delete

  • VirtualMachine.Provisioning.Clone

  • VirtualMachine.Interact.PowerOn

  • VirtualMachine.Config.AddRemoveDevice

When to use the Migration Forecaster

Use the Migration Forecaster when:

  • The default 620 Mbps transfer rate assumption does not reflect your actual network and storage performance.

  • You are planning migration for a large estate where time estimate accuracy is important for scheduling decisions.

  • You want to compare throughput across different source-and-target datastore combinations before committing to a migration topology.

Do not run the forecaster during a production maintenance window, as temporary VM creation in vCenter could affect other workloads running at the same time.

Note

The Migration Forecaster is available in specific OpenShift migration advisor builds. Check the in-product help and your release notes to confirm whether this feature is available in your environment.

Understanding the automated report

This chapter helps you read risk and readiness information, capacity-style outputs, and, when your build offers them, time-and-complexity-style views.

Interpreting risks and migration considerations in an automated migration report

When an automated migration report is ready, the Red Hat Hybrid Cloud Console and, where your build offers it, a local appliance, display findings the product derives from the data. These may be labeled as risks, blockers, warnings, or considerations depending on the release.

  • What the result is — A machine-driven view of items that match rules in the product and, where your version uses them, in optional policy engines such as OPA for virtual machine validation. The report does not replace a full human review of application behavior or of data and compliance needs.

  • What you should do — Triage each line with the owner of the workload and its dependencies. If the user interface links to a remediation note or a Red Hat document, use that as the source for the pattern the product detected, not a third-party copy.

  • What the result is not — The automated report is not a guarantee that every workload will run on OpenShift Virtualization or OpenShift without design or change work, or that all edge cases are covered.

Treating a message as a formal security or compliance finding is a decision for your own governance process beyond what the tool reports.

Sizing and capacity-related outputs

An automated migration report can include sizing guidance for a target OpenShift or OpenShift Virtualization footprint, in the form the product offers in your version. The service compares the aggregate resource picture from your RVTools file or agent run to the overhead and planning assumptions built into that release. It is not a substitute for measuring real utilization and storage performance on the cluster you eventually build, or for your resiliency and high-availability design.

  • How to use the output — Compare the suggested size to your own rules for headroom, maintenance windows, and separate failure domains. If you split work across many OpenShift projects or namespaces, you might need more nodes or pools than a single top-level line item shows.

  • When to investigate further — If very uneven resource use, unusual storage or network patterns, or traditional guests that are hard to size from inventory alone are indicated in the report, treat the sizing line as a preliminary estimate.

Final hardware or cloud shape and storage classes are decisions you make at detailed design, with your Red Hat account or services team and your own internal review when applicable.

Migration time and complexity in the product

Some of the OpenShift migration advisor builds may show a migration time estimate, a complexity view, or other planning hints next to sizing and risk information. The exact labels and controls come from the console. Do not treat a time range or a score as a service level agreement, a project deadline, or a full picture of work for every application team.

Important

Migration time-and-complexity-style outputs are partial or still evolving in many builds. They are not as mature as core inventory and risk views. The product may add, move, or hide them between updates. Your product release notes and the in-product help for your entitlement are the source of truth. If a field is missing, experimental, or labeled as beta in the user interface, do not use it for contractual planning.

When you do see a time or complexity line and you use it only for internal order-of-magnitude planning, still compare the result to your runbooks, maintenance windows, and application dependencies. The model does not account for blackout dates or data migration work outside the scope of the inventory.

After the automated report

This chapter places the OpenShift migration advisor next to migration toolkit for virtualization in a migration program, and suggests clear next steps once you have a result. It does not include partner or third party marketplaces.

How the OpenShift migration advisor relates to Migration Toolkit for Virtualization in a migration program

The OpenShift migration advisor and Migration Toolkit for Virtualization have different jobs in a typical program that moves virtual machines from VMware to OpenShift Virtualization. The OpenShift migration advisor is for a pre-migration automated migration report and planning view of the estate. Migration Toolkit for Virtualization is the Red Hat product line for executing the migration and managing the movement of virtual machines after you have a design and a target cluster that your organization has approved.

  • Use the OpenShift migration advisor when you need a structured inventory and readiness view and planning outputs from an RVTools file in a supported format or assisted agent run, as your build of the product allows.

  • Use Migration Toolkit for Virtualization when you are ready to create migration plans and to run cutover in line with the Migration Toolkit for Virtualization product documentation and your own change and security process.

A fully automated handover from a finished result from the OpenShift migration advisor into a Migration Toolkit for Virtualization plan is not guaranteed in every Red Hat offering, OpenShift and OpenShift Virtualization version, or subscription. Read the product documentation for the OpenShift migration advisor, OpenShift Virtualization, and Migration Toolkit for Virtualization for your environment and use only the integrated workflows that are documented by Red Hat for your release.

What to do after the automated report

An automated migration report is an input to a larger program, not the end of the work. After you are satisfied with the data that you submitted and the result in the Red Hat Hybrid Cloud Console, you can do the following with your own organization and, if you have a contract, with Red Hat:

  • Align owners — Share the result with the platform, application, and security owners who will sign off on a target design.

  • Close gaps — Address any high-severity items the report lists before you invest in a full Migration Toolkit for Virtualization cutover plan. Some items will need a human decision even when the tool cannot automate a fix.

  • Design the target — Build a final OpenShift and OpenShift Virtualization design that includes networks, storage, identity, and Day 2 operations. The automated report supports that design; it does not replace it.

  • Engage Red Hat — If you have a Red Hat account or support contract, use the support and services paths that are documented by your account team. Internal contact details or nonpublic programs are not included here.

Questions and answers about the OpenShift migration advisor

Common questions about the OpenShift migration advisor, including release cadence, language availability, and support for large VMware estates.⁠

What is the OpenShift migration advisor?

The OpenShift migration advisor is a self-service tool designed to evaluate the migration readiness of your VMware workloads before you commit. By discovering your vCenter environment, an automated migration report is generated that provides the technical details that you need to plan your journey with confidence. With the OpenShift migration advisor, you can assess your virtual machine workload migration path as you plan your move to OpenShift.

Who can access the OpenShift migration advisor, and how is it accessed?

You access the OpenShift migration advisor through the Red Hat Hybrid Cloud Console. Anyone with a Red Hat account can use the tool; you do not need a Red Hat subscription.

What types of insights does the OpenShift migration advisor provide?

  • Discovery — A clear snapshot of your connected VMware clusters, including CPU and memory, disks, networks, and related inventory.

  • Warnings and risks — Identification of potential obstacles that could affect your migration, such as specific hardware configurations or guest operating system versions. Future releases might add more remediation guidance when it is available.

  • Recommendations — Technical guidance on how workloads can be configured in a OpenShift environment, plus estimates such as migration time windows and complexity views if they are displayed in your environment, based on the inventory and rules that are available in your version.

How is the OpenShift migration advisor different from the Virtualization Migration Assessment (VMA)?

VMA is a Red Hat consulting engagement with workshops and architects for large or complex migration programs. The OpenShift migration advisor is an automated report in the Red Hat Hybrid Cloud Console. Teams often run the OpenShift migration advisor first to establish a baseline, then use VMA or other programs to operationalize.

How is the OpenShift migration advisor different from RVTools?

RVTools exports vSphere inventory to a spreadsheet. The OpenShift migration advisor consumes an RVTools export in a supported format or agent-collected inventory and applies rules and analytics to produce migration-oriented views in an automated migration report.

What are the two ways you can create an automated migration report?

You can create an automated migration report by uploading a supported RVTools export or by deploying the assisted migration agent in your VMware vSphere environment. Your environment might offer one or both paths. See RVTools upload compared with a VMware agent for a comparison.

How much does the OpenShift migration advisor cost?

The OpenShift migration advisor is a self-service tool in typical offerings. Console availability and labels follow your Red Hat entitlement or program. For navigation details, see How you use the OpenShift migration advisor in the Hybrid Cloud Console.

What is the target audience for this solution?

The OpenShift migration advisor is intended for VMware administrators, IT architects, and platform architects who are familiar with their vSphere environments and are evaluating OpenShift as a migration target. You must be able to generate or export the data that the service accepts, and you must understand what you are authorizing if you use an agent in your environment.

How does this help my migration planning?

The automated migration report gives you inventory-backed data, risk signals, and sizing context before you commit to a migration design or project timeline. It helps you identify potential issues early and establish a baseline for your migration program. See Problems that an automated migration report is meant to help with for more detail.

When should we use this in our evaluation process?

During discovery and evaluation—before you finalize designs or execution—when administrators and architects want inventory-backed signals without committing to a full migration plan yet. See also Problems that an automated migration report is meant to help with.

What documentation and learning materials are available?

Beyond this guide: Red Hat blogs and recordings where your program publishes them. Your account team or partner can share decks under your agreement.

Glossary and additional resources

This chapter defines terms used in this user guide and lists public kubev2v repositories for readers who work with source at a high level. It does not replace the Red Hat product documentation for a given release.

Glossary of terms for this guide

Short definitions to align language in the rest of this user guide. Product names and feature labels in the live console are authoritative for a given release.

  • Assisted migration agent — Software, distributed as an OVA or other supported image, that you deploy within your vSphere environment to collect inventory and send it to the reporting service according to the mode and consent settings you select.

  • Connected mode — An agent deployment mode in which the assisted migration agent has network access to the OpenShift migration advisor service endpoints and sends data directly. Contrast with disconnected mode.

  • Disconnected mode — An agent deployment mode in which the assisted migration agent operates without direct network access to OpenShift migration advisor service endpoints. Data is collected locally and transferred by another method. Contrast with connected mode.

  • kubev2v — The GitHub organization that publishes the open source components behind OpenShift migration advisor, including the Migration Planner service, the console UI, and the assisted migration agent. Red Hat support and product scope follow your subscription, not the public repositories.

  • Automated migration report — A result from the OpenShift migration advisor that is built from a supported data set and the rules and analytics in the product. It is a planning view, not a live migration.

  • Migration Planner — The Red Hat backend service that receives inventory data from the assisted migration agent or an RVTools upload, stores automated migration reports, and makes results available in the Red Hat Hybrid Cloud Console. The term also refers to the open source project of the same name in the kubev2v organization; the product and the repository are related but not identical in scope or support.

  • Migration Toolkit for Virtualization — The Red Hat software used to execute migration of virtual machines to OpenShift Virtualization. It is for migration execution, not a substitute for a pre-migration automated migration report, though you may use both in the same program of work.

  • OPA (Open Policy Agent) — An open source policy engine that some OpenShift migration advisor builds use to validate virtual machines against a set of rules before including them in an automated migration report. If your build requires OPA policy folders, the product documentation for your version describes how to supply them.

  • OVA (Open Virtualization Appliance) — A standard packaging format for virtual machine images. The assisted migration agent is distributed as an OVA that you import into your VMware environment.

  • RVTools — A free Windows utility from Dell Technologies that exports VMware vSphere inventory data to a spreadsheet. OpenShift migration advisor can accept an RVTools export in a supported format when that path is enabled in your product build.

  • vCenter — The VMware management platform that the assisted migration agent and RVTools both connect to in order to collect inventory. OpenShift migration advisor uses vCenter as the scope boundary for an automated migration report.

  • vSphere estate — The full set of VMware infrastructure — virtual machines, hosts, clusters, datastores, and networks — managed under a vCenter deployment. Used in this guide to refer to the VMware environment that an automated migration report targets.

Open source projects that implement the automated reporting stack

The OpenShift migration advisor on the Red Hat Hybrid Cloud Console is a Red Hat product. The kubev2v community on GitHub publishes source code for related components. Red Hat support, scope, and build identifiers follow your subscription and your product version, not a public repo alone.

  • migration-planner — Migration Planner service in the kubev2v organization.

  • migration-planner-ui-app — Console user interface in the kubev2v organization.

  • assisted-migration-agent — Assisted migration agent in the kubev2v organization.

If you open a public issue or a fork, remove host names, account identifiers, and credentials from logs or command output before you post, in line with your information security process.
